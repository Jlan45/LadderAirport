# Panel 前端优化方案

**日期：** 2026-07-23  
**状态：** 本轮修复已实施，持续优化项待排期  
**范围：** `web/` React 前端、与前端契约直接相关的 Panel API，以及 `panel/web/dist` 嵌入产物

## 1. 背景与目标

LadderAirport Panel 是面向日常运维的控制台，不是展示型站点。前端优化的首要目标是让节点巡检、入站配置、订阅维护和系统设置具备清晰、可恢复、可验证的操作闭环。

本方案聚焦以下结果：

1. 用户在任意页面都能明确看到加载、空数据、失败、处理中和成功状态。
2. 刷新、轮询、日志流和提交请求不会因并发或页面切换污染当前视图。
3. 核心流程在桌面和手机上均可完成，表格和抽屉不会溢出或遮挡操作。
4. 表单支持创建与编辑，客户端校验与服务端契约一致，敏感字段不会被意外覆盖。
5. 前端源码、测试结果和 Go 二进制内嵌静态资源保持一致。
6. 新增功能必须通过类型检查、单元测试、生产构建和关键流程冒烟测试。

## 2. 现状评估

### 2.1 信息架构

当前认证后的主导航已收敛为四个稳定入口：

| 导航 | 路由 | 核心任务 |
|------|------|----------|
| 节点 | `/` | 查看机群状态、筛选和选择节点、批量操作、打开节点详情 |
| 入站 | `/inbounds` | 创建、编辑、校验和启停协议入站 |
| 订阅 | `/subscriptions` | 管理订阅、选择本地入站、聚合外部订阅源 |
| 设置 | `/settings` | 维护刷新、探测、升级和安全相关配置 |

节点详情采用 `/?node=<id>` 驱动的侧边抽屉。该方式保留列表上下文，同时允许刷新和分享深链。独立批量页面已由节点页工具栏取代，避免同一批节点在多个页面重复管理。

### 2.2 已修问题

| 范围 | 原问题 | 已实施修复 | 用户收益 |
|------|--------|------------|----------|
| 应用壳层 | 页面一次性打入主包，认证失效处理分散 | 路由懒加载；统一监听 401；登录保留 `next`；增加退出登录和全局错误边界 | 首屏负担降低，会话失效后可预测地返回登录页 |
| 登录 | Enter 可能触发两次提交；错误信息不明确 | 单一表单提交链路、提交防重、401 中文提示、登录后返回原目标页 | 避免重复请求和操作上下文丢失 |
| 节点总览 | 首次加载、轮询、批量任务和单节点操作共用状态；旧请求可能覆盖新请求 | 分离各类状态；轮询防重；使用请求代次防止过期响应写回；删除节点后清理选择 | 状态反馈准确，快速切换或弱网下不再回跳 |
| 节点展示 | `pending`、`unknown` 被误判为离线；缺失指标显示为 `0`；任务状态中英文混杂 | 集中状态映射；未知指标显示 `—`；任务状态中文化 | 避免误判节点健康和资源使用情况 |
| 节点详情 | 切换节点时详情或日志可能串写；连接配置缺少完整校验 | 请求和日志流绑定节点 ID/代次；连接字段、标签、Token 按字段编辑；错误、加载和日志跟随状态独立呈现 | 抽屉可恢复，敏感字段不会因普通编辑被覆盖 |
| 添加节点 | 主机、端口和标签校验不足；忙碌时可关闭；HTTP 下复制失败 | 增加字段校验和标签去重；提交中锁定关闭；剪贴板增加兼容回退 | 降低无效配置和安装命令复制失败率 |
| 入站 | 只支持创建，编辑和启停链路不完整；动态字段语义不足 | 补齐编辑、启停和校验；动态表单增加标签、错误关联、端口范围和编辑值提取 | 入站全生命周期可在一个页面完成 |
| 订阅 | 本地订阅与外部源操作不完整；无法明确表达“全部本地入站” | 重构列表和编辑对话框；支持外部源编辑、纯外部订阅、请求头 JSON、预览、逐项忙碌状态；新增 `include_all_inbounds` 契约 | 订阅组合含义明确，编辑不会误选全部入站 |
| 设置 | 输入值直接等同于已保存值；失败或离开时状态不清楚 | 分离草稿与已保存状态；增加脏数据提示、放弃确认、URL/数字/密码校验和加载错误 | 用户能判断哪些改动尚未生效 |
| API 契约 | Node 更新会覆盖未提交字段；布尔值无法区分缺省和 `false`；错误体格式不统一 | Node 使用局部更新；Inbound `enabled` 使用指针语义；客户端统一解析 JSON/文本错误；补齐 logout | 编辑行为与用户提交内容一致 |
| 响应式 | 工具栏、表格、抽屉和紧凑操作在窄屏易拥挤 | 工具栏换行；表格横向滚动；抽屉宽度限制为视口；小屏调整操作区和字段布局 | 手机仍可完成查看与维护任务 |
| 可访问性 | 缺少跳转主内容、表单关联和图标按钮名称 | 增加跳转链接、语义导航、页面标题、字段标签、`aria-label` 和图标提示 | 键盘和辅助技术用户可识别主要结构与操作 |

### 2.3 仍需持续治理的风险

| 风险 | 影响 | 治理方向 |
|------|------|----------|
| 交互测试覆盖仍偏低 | 重构对话框、抽屉或异步状态时容易回归 | 引入组件测试和端到端关键流程测试 |
| 生产包只有构建告警，没有固定预算 | 依赖升级可能重新造成首包膨胀 | 在 CI 中记录并限制入口和路由 chunk 大小 |
| 真实浏览器兼容验证依赖人工 | 移动端、焦点和复制回退可能漏测 | 建立 Playwright 桌面/移动端冒烟矩阵 |
| 表格数据量增长后仍是全量渲染 | 大规模节点或入站下滚动和筛选变慢 | 达到阈值后增加服务端分页或虚拟列表 |
| 错误消息仍主要依赖即时提示 | 消息消失后不利于定位失败项 | 关键区域保留可重试的内联错误，任务保留最近结果 |

## 3. 信息架构与核心流程

### 3.1 页面层级

```text
登录
└── Panel 应用壳层
    ├── 节点总览
    │   ├── 筛选 / 视图切换 / 批量操作
    │   ├── 添加节点对话框
    │   └── 节点详情抽屉（?node=<id>）
    ├── 入站
    │   └── 创建 / 编辑动态协议表单
    ├── 订阅
    │   ├── 订阅编辑
    │   └── 外部源编辑 / 预览
    └── 设置
        └── 草稿编辑 / 保存 / 放弃
```

### 3.2 关键任务闭环

| 任务 | 标准流程 | 完成判定 |
|------|----------|----------|
| 新增并部署节点 | 添加节点 → 复制安装命令 → 节点上线 → 打开详情 → 关联入站 → 下发 | 总览显示节点在线且任务成功，详情指标可读取 |
| 调整入站 | 编辑参数 → 本地校验 → 保存 → 选择受影响节点 → 下发 | 入站保存成功，批量任务逐节点给出结果 |
| 发布订阅 | 选择全部或指定本地入站 → 可选外部源 → 预览 → 保存 → 复制链接 | 预览与保存后的实际订阅范围一致 |
| 修改设置 | 编辑草稿 → 字段校验 → 保存，或放弃改动 | 保存后脏状态消失；失败时草稿保留并可重试 |
| 排查节点 | 从总览打开抽屉 → 查看连接/指标/配置 → 启动日志流 → 执行操作 | 日志和操作结果始终属于当前节点，关闭抽屉后请求被清理 |

### 3.3 路由规则

- 未认证访问业务路由时跳转 `/login?next=<原路径>`，登录后返回原位置。
- 全局收到 401 时只触发一次会话失效流程，不在各页面重复实现跳转。
- `?node=<id>` 是节点详情的唯一可分享状态；关闭抽屉只移除该参数，不丢失列表筛选和视图状态。
- 无效节点深链必须显示明确错误和关闭/重试操作，不能永久停留在“加载中”。
- 未知路径回到节点页；页面标题随当前导航更新。

## 4. 交互与视觉原则

### 4.1 状态必须具体

不要用一个 `loading` 表示整页所有动作。每个异步边界至少区分：

- 首次加载：使用稳定占位或居中加载状态，不闪现空数据。
- 后台刷新：保留已有内容，仅在刷新按钮或局部区域提示。
- 单项提交：只锁定对应行或按钮，不阻塞无关操作。
- 批量任务：显示总进度和逐节点结果，部分成功不等同于全部失败。
- 空数据：解释当前结果为空，并提供与当前任务直接相关的入口。
- 失败：在失败区域保留错误和重试动作；瞬时消息只作为补充。

### 4.2 操作必须可预期

- 同一提交按钮在请求完成前不可重复触发。
- 删除、覆盖敏感配置、放弃未保存修改等不可逆操作需要确认。
- 关闭对话框、抽屉或切换节点时中止流式请求，并阻止过期响应更新新视图。
- 保存成功后更新本地已保存快照；保存失败保留用户输入。
- 图标用于刷新、复制、关闭、显隐等熟悉动作，并提供 Tooltip 和无障碍名称。
- 开关用于启停，分段控件用于互斥视图，菜单用于选项集；不要用样式相似的文本按钮混淆语义。

### 4.3 运维界面保持高信息密度

- 页面使用清晰的标题、工具栏和数据区域，不把每个分区包装成装饰卡片。
- 节点状态、运行状态和任务状态使用一致的词汇及颜色，颜色不能是唯一信息载体。
- 未获取到的数据用 `—`，真实数值为零时才显示 `0`。
- Token、密码和请求头等敏感信息默认遮蔽；复制与显隐是显式动作。
- 中文术语保持统一：节点、入站、订阅、外部源、下发、探测、部分成功。

## 5. 技术优化方案

### 5.1 性能

| 措施 | 当前状态 | 验收标准 |
|------|----------|----------|
| 路由级代码拆分 | 已实施 | 登录及四个业务页面使用动态加载；构建不出现单 chunk 超过 500 KiB 的告警 |
| 避免重复刷新 | 已实施 | 轮询未结束时不启动下一轮；手动刷新与自动刷新不会并发覆盖 |
| 局部更新 | 已实施 | 单行操作不触发整页遮罩；详情更新后只刷新必要数据 |
| 生产包预算 | 待加入门禁 | 入口 JS gzip 建议不超过 250 KiB；任一路由 chunk 原始大小不超过 500 KiB；超限需在变更说明中给出原因 |
| 大列表策略 | 按规模触发 | 任一列表超过 500 行或交互明显掉帧时，采用服务端分页或虚拟列表，避免同时渲染全部行 |
| 资源缓存 | 后续增强 | 带 hash 的静态资源长期缓存，HTML 和 API 禁止错误的长期缓存 |

性能验证至少记录一次生产构建产物大小，并在桌面与移动网络节流下检查登录后首个可操作时间。当前阶段不为了微小收益引入额外状态管理库。

### 5.2 可靠性

所有可能跨越组件生命周期的请求遵循同一规则：

1. 发起请求时捕获资源 ID 和递增代次。
2. 响应写入前确认组件仍挂载、资源 ID 未变化且代次仍是最新值。
3. 新请求、切换资源或卸载时中止可取消请求。
4. 流式日志关闭后不再接收行，也不把旧节点日志写入新节点。
5. 错误转换集中在 API 客户端，页面只决定错误展示位置。

可靠性验收场景：

- 连续快速切换两个节点，最终详情和日志只属于最后一个节点。
- 在慢请求期间关闭抽屉，控制台无卸载后更新警告。
- 自动刷新与手动探测同时触发，旧结果不能覆盖新结果。
- 会话过期时从任意页面进入同一登录流程，重新登录可返回原页面。
- 后端返回 JSON、纯文本或空错误体时，页面均显示可理解的错误。

### 5.3 表单与 API 契约

- 创建和更新输入类型分离；更新请求只发送用户实际编辑的字段。
- 需要区分“未提交”和零值的字段在后端使用指针或等价的可选语义。
- 客户端校验覆盖必填、端口范围、URL、JSON、重复标签和字段间依赖；服务端继续执行最终校验。
- 协议动态表单由模板 schema 驱动，但字段标签、帮助和错误必须关联到具体控件。
- Token 仅在用户明确修改时发送；清空敏感字段必须是可识别的显式操作。
- 订阅用 `include_all_inbounds` 明确区分“全部本地入站”和“未选择本地入站”，兼容旧数据时由服务端迁移并测试。

### 5.4 可访问性

| 项目 | 要求 | 验收方式 |
|------|------|----------|
| 页面结构 | 主导航、主内容、标题层级和跳转主内容链接语义完整 | 键盘 Tab 首次可进入跳转链接；读屏可识别导航与主区域 |
| 键盘操作 | 所有按钮、开关、表单、对话框和抽屉无需鼠标即可完成 | 仅使用 Tab、Shift+Tab、Enter、Space、Escape 走完核心流程 |
| 焦点管理 | 打开浮层时焦点进入，关闭后回到触发元素；错误后聚焦首个无效字段 | 手工测试添加节点、编辑入站、订阅和节点抽屉 |
| 控件名称 | 图标按钮必须有 `aria-label`，字段有可见标签，错误通过 `aria-describedby` 关联 | 浏览器无障碍树检查；后续加入 axe 自动扫描 |
| 状态通知 | 保存结果、任务状态和加载变化可被辅助技术感知 | 关键状态使用适度的 `aria-live`，避免重复播报轮询 |
| 颜色与对比度 | 状态不能只依赖颜色；正文、边框和焦点环满足 WCAG 2.1 AA | 对比度检查，焦点环在明暗背景上均可见 |

### 5.5 响应式

建议以内容是否溢出为依据维持现有小屏断点，并固定以下行为：

| 视口 | 布局要求 |
|------|----------|
| `>= 1024px` | 顶栏完整展示；节点表格和详情抽屉可并列观察；工具栏保持紧凑 |
| `641px - 1023px` | 工具栏允许换行；次要文字可缩短；表格区域独立横向滚动 |
| `<= 640px` | 导航和操作保持可触达；表单单列；对话框和抽屉接近全宽；主要按钮不被底部遮挡 |
| `<= 420px` | 隐藏非必要品牌文字；长 Token、URL、地址和任务错误允许断行，不撑破容器 |

响应式验收视口至少包括 `1440×900`、`1024×768`、`768×1024`、`390×844` 和 `320×568`。每个视口检查：无页面级横向滚动、控件文字不截断、浮层不越界、表格可滚动、固定区域不遮挡最后一项。

### 5.6 安全与隐私

- 生产环境使用 HTTPS；HTTP 剪贴板回退只解决兼容性，不替代传输安全。
- 不把 Token、密码、请求头或 CA 私钥放入 URL、日志、错误堆栈和持久化视图状态。
- 退出登录调用服务端注销并清理客户端认证状态；401 统一收敛。
- 用户粘贴的请求头 JSON 只解析为结构化数据，不使用动态执行或 HTML 注入。
- 错误边界展示可恢复说明，详细错误仅写入开发者控制台，避免向终端用户暴露内部实现。

## 6. 测试策略

### 6.1 测试分层

| 层级 | 覆盖对象 | 当前 / 下一步 |
|------|----------|---------------|
| 类型检查 | API 类型、组件属性、可选字段与空值处理 | 已有 `tsc -b`，作为构建前置门禁 |
| 纯函数单元测试 | 状态文案、字节和时间格式化、任务状态映射 | 已有 Node Test；继续覆盖边界值和未知状态 |
| API/存储测试 | 局部更新、布尔缺省语义、订阅迁移、logout | Go 测试覆盖，契约变更必须同步增加回归用例 |
| 组件测试 | 表单校验、脏状态、按钮防重、加载/错误/空状态 | P1 引入 Vitest + Testing Library，优先覆盖高风险交互 |
| 端到端测试 | 登录、节点深链、入站编辑、订阅预览、设置保存 | P1 引入 Playwright，桌面和移动端运行最小冒烟集 |
| 视觉检查 | 溢出、遮挡、断点、对话框和抽屉尺寸 | 发布前保留关键视口截图；稳定后做基线差异检查 |

### 6.2 必测回归清单

- 登录：Enter 只提交一次；错误密码有中文提示；`next` 回跳正确；退出后受保护页面不可见。
- 节点：首次加载、刷新失败、空列表、筛选、全选可见项、批量部分成功、删除后选择清理。
- 节点详情：有效和无效深链、快速切换、连接字段编辑、Token 保持不变/明确修改、日志开始/停止/跟随。
- 入站：所有协议创建和编辑；端口边界；必填错误；启停；凭据复制。
- 订阅：全部/指定/无本地入站；仅外部源；请求头 JSON 错误；预览失败；单项启停不锁全页。
- 设置：加载失败；无修改保存禁用；无效 URL/数字/密码；放弃确认；保存失败保留草稿。
- 响应式和键盘：上述核心流程至少在一个桌面和一个移动视口完成一次。

## 7. 构建与发布门禁

### 7.1 本地验证顺序

```bash
cd web
npm ci
npm run typecheck
npm test
npm run build

cd ../panel
go test ./...

cd ../pkg
go test ./...

cd ../agent
go test -tags with_quic,with_utls ./... -timeout 120s
```

`npm run build` 已包含类型检查；单独执行 `npm run typecheck` 是为了更快定位类型错误。依赖未变化时，本地开发可用 `npm install` 代替重复的干净安装，但 CI 必须使用 `npm ci`。

### 7.2 内嵌静态资源一致性

Panel 构建使用提交在 `panel/web/dist` 中的前端产物。每次修改 `web/src` 后必须：

1. 在 `web/` 完成生产构建。
2. 将 `web/dist` 同步到 `panel/web/dist`。
3. 执行一致性检查：

```bash
diff -qr web/dist panel/web/dist
```

任何差异都应阻断合并，防止源码已修复但发布二进制仍携带旧界面。

### 7.3 CI 合并条件

- `npm test`、`npm run build` 全部通过，无 TypeScript 错误。
- `web/dist` 与 `panel/web/dist` 完全一致。
- `panel`、`pkg`、带 `with_quic,with_utls` 标签的 `agent` 测试通过。
- `git diff --check` 无空白错误。
- 新增 API 字段同时具备存储迁移、旧数据兼容和 handler 回归测试。
- P0/P1 前端缺陷修复必须附对应自动化测试；暂时无法自动化时，在变更说明中记录手工步骤和结果。
- 发布候选版本完成桌面和移动端关键流程冒烟，不存在阻塞操作的控制台异常、重叠或空白页面。

## 8. 可观测性与问题定位

前端不应吞掉失败，也不应把所有失败压缩成“请求失败”。建议逐步统一以下上下文：

- API 错误保留 HTTP 状态、后端消息和请求类别，不在 UI 暴露敏感响应体。
- 批量任务保留最近一次任务类型、开始时间、总数、成功数、失败数和逐项原因。
- Error Boundary 提供重新加载入口，并在开发环境输出组件堆栈。
- 日志流区分“用户停止”“节点切换”“网络中断”和“服务端错误”。
- 若后续接入遥测，优先记录页面加载失败率、API 失败率、批量任务失败率和前端未捕获异常；不得采集 Token、密码或订阅内容。

## 9. 优先级路线图

### P0：本轮发布前收口

| 工作 | 交付物 | 完成标准 |
|------|--------|----------|
| 节点详情竞态和错误态 | 抽屉请求代次、日志中止、无效深链、字段校验 | 快速切换和关闭抽屉无串数据；错误可关闭或重试 |
| 全量类型与构建修复 | `typecheck`、测试、生产构建 | 所有命令通过，无未使用变量和无效属性类型 |
| 后端契约复核 | Node/Inbound/Subscription/logout 测试 | 局部更新、缺省布尔和旧订阅迁移均有回归测试 |
| 内嵌产物同步 | 两份 `dist` 内容一致 | `diff -qr` 无输出，CI Web 检查通过 |
| 多视口冒烟 | 桌面/移动截图与操作记录 | 核心页面无重叠、越界、空白或不可触达操作 |

### P1：下一迭代，降低回归率

| 工作 | 价值 | 完成标准 |
|------|------|----------|
| 建立组件测试栈 | 覆盖复杂表单和异步状态 | Fleet、NodeDetail、Inbounds、Subscriptions、Settings 各有关键交互测试 |
| Playwright 冒烟 | 自动验证真实浏览器行为 | 登录和四条核心任务在 Chromium 桌面/移动项目通过 |
| 可访问性审计 | 补齐焦点和播报问题 | 键盘走通核心流程，自动扫描无严重违规 |
| 包体预算门禁 | 防止依赖升级导致性能倒退 | CI 输出各 chunk 大小并对预算超限失败 |

### P2：中期体验增强

| 工作 | 触发条件 / 目标 |
|------|-----------------|
| 分页或虚拟列表 | 节点、入站或订阅达到数百项并出现明显渲染延迟 |
| 统一任务中心 | 批量下发、探测、启停和升级需要跨页面持续查看时 |
| 筛选状态 URL 化 | 运维人员需要分享或恢复复杂列表视图时 |
| 细化权限和审计 | Panel 出现多用户、只读角色或变更追踪需求时 |
| 前端遥测 | 部署规模扩大，需要量化异常率和关键操作成功率时 |

### P3：长期治理

- 建立设计令牌和可复用的页面状态组件，但只抽象已经重复且稳定的模式。
- 对依赖升级执行可访问性、包体和视觉回归检查。
- 按真实使用数据调整轮询间隔、列表密度和移动端操作顺序。
- 每个版本复盘前端故障，将新发现的失败模式转成自动化测试或发布门禁。

## 10. 完成定义

一次 Panel 前端改动只有同时满足以下条件才算完成：

1. 正常、加载、空数据、错误和处理中状态均有明确界面。
2. 快速重复操作、页面切换和慢网络不会产生过期数据或重复提交。
3. 桌面和移动端核心流程可完成，键盘可到达所有必要操作。
4. 类型检查、前端测试、生产构建和相关 Go 测试通过。
5. `web/dist` 与 `panel/web/dist` 已同步并由 CI 校验。
6. 用户可见行为变化有测试或可复现的手工验收记录。

该定义应作为后续 Panel 功能开发和缺陷修复的共同发布标准。
